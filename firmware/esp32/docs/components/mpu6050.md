## Overview

MPU6050 เป็น IMU หลักของ FallHelp ใช้ข้อมูล accelerometer และ gyroscope เพื่อตรวจแรงกระแทก, มุมเอียง, และ posture change ของผู้สวมใส่

เอกสารนี้เป็น owner doc ของ MPU6050 และ fall detection core ไม่ใช่ lab manual ของ Fall Detection Sensor Lab

| รายการ | ค่า |
| --- | --- |
| Component | MPU6050 Accelerometer + Gyroscope |
| Interface | I2C |

## Calibration

เป้าหมายหลัง calibrate:

1. `SVM = sqrt(ax^2 + ay^2 + az^2) ≈ 1.0g` ขณะอุปกรณ์นิ่ง
2. gyro drift ใกล้ 0 ขณะอุปกรณ์นิ่ง
3. ค่า offset สอดคล้องกับตำแหน่งสวมใส่จริง

เปิด calibration mode ใน `firmware/esp32/src/sensor_tuning/MPU6050_Sensor.ino`:

```cpp
#define SOFTWARE_CALIBRATION_MODE true
```

ขั้นตอน:

1. คล้องอุปกรณ์ที่คอในตำแหน่งใช้งานจริง
2. ยืนนิ่งตรง
3. รอ warmup countdown 3 วินาที
4. ให้ firmware เก็บ 400 samples
5. copy ค่า `MPU_CAL_*` ที่พิมพ์ออกมาไปใส่ code
6. เปลี่ยน `SOFTWARE_CALIBRATION_MODE` กลับเป็น `false`

ค่า calibration ที่ lock อยู่ใน firmware ปัจจุบัน:

```cpp
#define MPU_CAL_ACCEL_OFFSET_X_MS2  1.127f
#define MPU_CAL_ACCEL_OFFSET_Y_MS2  2.158f
#define MPU_CAL_ACCEL_OFFSET_Z_MS2 -0.036f
#define MPU_CAL_GYRO_OFFSET_X_RAD  -0.061f
#define MPU_CAL_GYRO_OFFSET_Y_RAD   0.011f
#define MPU_CAL_GYRO_OFFSET_Z_RAD  -0.020f
```

ค่าเหล่านี้ใช้ทั้ง `main_firmware` และ `sensor_tuning` อย่าแก้เว้นแต่ตำแหน่งติดตั้งเปลี่ยนหรือมีรอบ calibration ใหม่ที่มีหลักฐานครบ

## swCal vs hwOff

| รายการ | swCal | hwOff |
| --- | --- | --- |
| ที่เก็บค่า | `MPU_CAL_*` ใน source | hardware register ใน MPU6050 |
| จุดที่ apply | หลังอ่าน I2C ใน firmware | ก่อนส่งค่าออกจาก IC |
| ใช้ใน FallHelp | ใช้ | ไม่ใช้ |

FallHelp ตั้งใจใช้ software calibration เพื่อให้ค่าที่ใช้ตรวจสอบและ sync ระหว่าง firmware variants ได้ง่าย

## Runtime Behavior

Fall detection core:

```text
read accel + gyro
  -> complementary filter computes pitch/roll
  -> SVM detects impact magnitude
  -> postureDelta checks posture change after impact
  -> duration gate confirms timing
  -> suspected_fall
  -> cancel window
  -> fall_cancelled or fall_confirmed
```

ค่าหลักใน firmware ปัจจุบัน:

| ค่า | ปัจจุบัน | ใช้ทำอะไร |
| --- | --- | --- |
| Impact threshold | `2.0g` default | gate แรงกระแทกจาก SVM |
| Duration threshold | `1500 ms` default | gate เวลากระแทก/รอ stabilization |
| Posture threshold | `45 deg` default | gate การเปลี่ยนท่าหลัง impact |
| Cancel timeout | `15000 ms` | รอผู้สวมใส่กด cancel หลัง `suspected_fall` |

`main_firmware` ใช้ cancel window คงที่จาก `FallDetectionConfig.ino` และไม่ใช้ runtime profile แยกแล้ว

## SVM And Posture Delta

SVM:

```text
SVM = sqrt(ax^2 + ay^2 + az^2)
```

ใช้ดู magnitude ของแรงลัพธ์จาก accelerometer โดยไม่สนใจ orientation ของอุปกรณ์

postureDelta:

```text
postureDelta = posture difference around impact window
```

ใช้แยกกิจกรรมที่แรงแต่ไม่ได้ล้มออกจาก fall-like posture change

ตัวอย่าง:

| กรณี | SVM | postureDelta | ผลที่คาด |
| --- | --- | --- | --- |
| ล้มข้าง | สูงกว่า threshold | สูงกว่า threshold | `suspected_fall` |
| นั่งลงแรง | อาจสูง | ต่ำ | ไม่ trigger |
| เดินเร็ว | อาจมี spike | ต่ำ | ไม่ trigger |
| ล้มเบาเกิน threshold | ต่ำ | ไม่พอใช้ตัดสิน | ไม่ trigger |

## Diagnostic Log vs Fall Detection Log

Diagnostic log:

```text
[abnormal] accel=(x,y,z)g gyr=(rx,ry,rz)rad/s swCal=(dx,dy,dz) fsRange=...
```

ใช้ดู calibration, sensor range, และค่าผิดปกติจาก IMU ไม่ใช่หลักฐานว่า firmware ตรวจพบการล้ม

Fall detection log:

```text
[fall] svm=X.XXg postureDelta=XX.Xdeg -> suspected_fall / cancel / confirm
```

ใช้ดู decision path ของ fall flow

## Serial Commands And Safety

คำสั่ง `mpu on` เปิด IMU diagnostic stream และทำให้ fall detection ถูกปิดในโหมดนั้น

| สถานะ | Fall detection | ใช้เมื่อ |
| --- | --- | --- |
| `mpu on` | ปิด | debug raw IMU เท่านั้น |
| `mpu off` หรือโหมดปกติ | เปิด | fall/ADL test และ runtime flow |

อย่าใช้ `mpu on` ระหว่าง fall/ADL session ที่ต้องการให้ firmware generate `suspected_fall` หรือ `fall_confirmed`

## Test Checklist

### Calibration Check

1. ใช้ `sensor_tuning`
2. เปิด `SOFTWARE_CALIBRATION_MODE true`
3. สวมอุปกรณ์ตำแหน่งจริง
4. เก็บ 400 samples
5. เลือกรอบที่ magnitude ใกล้ `1.0g` ที่สุด

### Fall Core Check

1. ใช้ firmware variant ที่ต้องการทดสอบ
2. ยืนยันว่าไม่ได้เปิด `mpu on`
3. รัน `sim fall` หรือรอบทดสอบที่กำหนด
4. ดู SVM, postureDelta, และ gate ที่ผ่าน/ไม่ผ่านจาก Serial log
5. ถ้าเป็น system flow ให้ยืนยัน `suspected_fall -> fall_cancelled / fall_confirmed`

### Fall Detection Sensor Lab Boundary Check

1. ใช้ `sensor_tuning` เท่านั้นสำหรับ Basic Activity Collection
2. ใช้ workflow รายละเอียดใน `fall_detection_sensor_lab/`
3. อย่าสรุปผลเชิงสถิติจากข้อมูลชุดนี้

## Troubleshooting

### ไม่ Detect Fall

ตรวจ:

1. `mpu on` ปิด fall detection อยู่หรือไม่
2. SVM ต่ำกว่า impact threshold หรือไม่
3. postureDelta ต่ำกว่า threshold หรือไม่
4. calibration offset ผิดจากตำแหน่งสวมใส่จริงหรือไม่
5. firmware variant และ build profile ถูกต้องหรือไม่

### Trigger บ่อยในกิจกรรมปกติ

ตรวจ:

1. calibration ทำให้ baseline SVM เบี้ยวหรือไม่
2. postureDelta threshold ต่ำเกินไปหรือไม่
3. อุปกรณ์ขยับบนคอมากเกินกว่าการใช้งานจริงหรือไม่
4. ปรับหลายค่าในรอบเดียวจนวิเคราะห์ย้อนกลับไม่ได้หรือไม่

### Cancel / Confirm Flow ไม่ครบ

ตรวจ:

1. button GPIO27 พร้อมใช้งานหรือไม่
2. cancel timeout ยังเป็น `15000 ms` หรือไม่
3. AlertSystem reset state ถูกจังหวะหรือไม่
4. MQTT connected อยู่หรือไม่ถ้าต้องเห็น event ฝั่ง backend

### Sensor Lab CSV ไม่ตรงที่คาด

ตรวจ:

1. ใช้ `sensor_tuning` และ topic `device/+/lab/imu` หรือไม่
2. Node-RED Dashboard อยู่ใน workflow Fall Detection Sensor Lab จริงหรือไม่
3. CSV schema และ trial rule ให้ตรวจจาก `fall_detection_sensor_lab/` ไม่ใช่ไฟล์นี้
